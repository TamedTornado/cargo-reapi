# Linux XFS reflink population statistics — 2026-07-20

Host: Ubuntu 24.04, Linux 6.8, x86_64, 20 logical CPUs, approximately
62.6 GiB RAM. The qualification data root was a sparse XFS loop volume on the
host's ext4 NVMe RAID1 filesystem, formatted with `reflink=1`. Tested
cargo-reapi revision: `8d5ec08`. Acceptance contract SHA-256:
`c833908214b7de8a7c593fee7799de7d1fbe5088b411b6d00fca7aa4ef4da500`.

These statistics are from the three Moria population proofs that passed before
the later exact-mutation experiment correctly rejected an over-broad rebuild.
Per project retention policy, the failed aggregate batch and its raw build
artifacts were deleted after these passing statistics were extracted. These
figures are benchmark statistics, not a complete Linux qualification receipt.

| Population | Elapsed | Performance reference | Snapshot hits | Physical cacheable actions | OS-observed compiler/linker actions | Correctness |
| --- | ---: | ---: | ---: | ---: | ---: | --- |
| One clean consumer | 6.939s | 60s | 3/3 | 0 | 0 | pass |
| Five simultaneous clean consumers | 10.680s | 120s | 15/15 | 0 | 0 | pass |
| Ten simultaneous clean consumers | 18.848s | 120s | 30/30 | 0 | 0 | pass |

Every target was empty at start, and every member in the five- and ten-member
populations started before any member completed. All consumers restored the
complete linked Moria target with Linux copy-on-write clones. The independent
OS observers recorded no compiler or linker execution in any warm population.

After all three populations, including ten simultaneous consumer targets, the
260 GiB sparse XFS volume had 35 GiB physically allocated and 226 GiB free.
The preceding ext4 portable-copy run exhausted its host filesystem before the
ten-member population could complete. The XFS result therefore demonstrates
that reflink removes the full-target storage amplification while retaining
independent writable worktrees.
